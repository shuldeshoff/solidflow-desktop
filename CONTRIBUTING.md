# Руководство по внесению вклада

Спасибо за интерес к проекту SolidFlow Desktop! Мы рады вашему участию.

## Как внести вклад

### Сообщение об ошибке

Если вы нашли ошибку:

1. Проверьте, что ошибка еще не описана в [Issues](https://github.com/shuldeshoff/solidflow-desktop/issues)
2. Создайте новый Issue с описанием:
   * Что произошло
   * Ожидаемое поведение
   * Шаги для воспроизведения
   * Версия Python и операционная система
   * Скриншоты (если применимо)

### Предложение новой функции

1. Проверьте, что функция еще не предложена в [Issues](https://github.com/shuldeshoff/solidflow-desktop/issues) или [Discussions](https://github.com/shuldeshoff/solidflow-desktop/discussions)
2. Создайте Issue с описанием:
   * Какую проблему решает эта функция
   * Предлагаемое решение
   * Альтернативные варианты (если есть)

### Внесение изменений в код

1. **Fork** репозитория
2. Создайте новую ветку от `main`:
   ```bash
   git checkout -b feature/название-функции
   # или
   git checkout -b fix/описание-исправления
   ```
3. Внесите изменения
4. Убедитесь, что код соответствует стандартам проекта:
   * Запустите форматирование: `black src/ tests/`
   * Проверьте линтинг: `flake8 src/ tests/`
   * Запустите тесты: `pytest`
5. Закоммитьте изменения с понятными сообщениями:
   ```bash
   git commit -m "feat: добавить новую функцию"
   # или
   git commit -m "fix: исправить баг в модуле X"
   ```
6. Запушьте изменения:
   ```bash
   git push origin feature/название-функции
   ```
7. Создайте **Pull Request** в репозитории

## Стандарты кода

### Стиль кода

* Используйте `black` для форматирования кода
* Максимальная длина строки: 100 символов
* Используйте `flake8` для проверки стиля

### Документация

* Добавляйте docstrings для всех публичных функций и классов
* Используйте Google или NumPy стиль docstrings
* Обновляйте документацию при изменении API

### Тестирование

* Пишите тесты для нового кода
* Убедитесь, что все тесты проходят
* Стремитесь к покрытию кода не менее 70%

### Коммиты

Используйте префиксы для сообщений коммитов:

* `feat:` - новая функция
* `fix:` - исправление бага
* `docs:` - изменения в документации
* `style:` - форматирование, отсутствующие точки с запятой и т.д.
* `refactor:` - рефакторинг кода
* `test:` - добавление или изменение тестов
* `chore:` - изменения в процессе сборки или вспомогательных инструментах

Примеры:
```
feat: добавить импорт STEP файлов
fix: исправить ошибку в анализе толщины стенок
docs: обновить README с инструкциями по установке
```

## Процесс ревью

1. Все Pull Requests проходят ревью кода
2. Убедитесь, что CI/CD проверки проходят
3. Отвечайте на комментарии ревьюеров
4. После одобрения изменения будут влиты в основную ветку

## Вопросы?

Если у вас есть вопросы, создайте [Discussion](https://github.com/shuldeshoff/solidflow-desktop/discussions) или [Issue](https://github.com/shuldeshoff/solidflow-desktop/issues).

Спасибо за ваш вклад!
